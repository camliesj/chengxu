# Web 与 Android 统一新增工单设计

日期：2026-07-21
状态：已批准
范围：完整业务能力阶段 2（新增工单纵向切片）

## 1. 背景与目标

阶段 1 已完成工单领域模型、权限与能力开关、幂等记录、兼容读取 API、Android 加密字段存储以及 Room v2 基础表。阶段 2 需要交付可实际使用的新增工单能力。

本阶段不采用 Android 专用创建规则。网页端与 Android 必须共享服务端创建流程、编号、字典、默认值、校验、权限和错误语义，避免两端正式业务数据不同步。Windows 或其他仍使用旧 `POST /api/orders` 的客户端通过兼容适配层进入同一创建服务。

目标：

- 网页端和 Android 使用同一业务契约创建工单；
- 服务端成为正式编号、企业归属、状态和时间的唯一可信来源；
- 通过 `operationId` 保证网络重试不会产生重复工单；
- 两端提供一致的四步创建流程与本地草稿恢复；
- 创建成功后立即以服务端结果刷新详情、列表和工作台；
- 保留公司隔离、角色权限、敏感字段保护和可关闭的生产能力开关。

非目标：

- 本阶段不实现编辑工单、推进状态、结算、回款、作废或导出；
- 本阶段不增加离线写入队列，离线只能编辑已存在的本地草稿；
- 不要求网页与 Android 像素完全一致，但要求字段、文字、状态和操作结果一致；
- 不启动 Android 模拟器，连接式测试仅保留源码并完成编译。

## 2. 核心架构决策

### 2.1 单一创建服务

新增语义接口：

```text
GET  /api/order-creation-metadata
POST /api/orders/create
GET  /api/order-operations/create-order/:operationId
```

网页端和 Android 的新增动作均调用 `POST /api/orders/create`。现有 `POST /api/orders` 同时承载旧客户端的新增与既有工单更新，所以不能在阶段 2 整体改成创建接口：当目标 ID 不存在、请求语义为新增时，兼容分支把旧载荷转换成新命令并调用同一创建服务；当目标工单已存在时，暂时保留当前更新路径，等待后续“编辑工单”阶段迁移到专用命令。兼容新增必须把既有 `eventId` 映射为 `operationId`，缺少幂等 ID 时拒绝不安全创建。旧新增请求中的 `id`、`companyId`、`role`、`status`、`version`、结算、回款和作废字段不能覆盖服务端创建字段。

共享服务模块负责：

1. 校验登录会话、企业和 `CREATE_ORDER` 能力；
2. 规范化并校验创建字段；
3. 执行幂等操作认领；
4. 生成正式编号和北京时间；
5. 写入工单并完成幂等回执；
6. 返回完整标准化工单、服务端时间与最新能力列表。

### 2.2 正式编号

正式工单号格式统一为：

```text
ROYYYYMM#####
```

例如 `RO20260700001`。`YYYYMM` 按北京时间取值，五位序号按月从 `00001` 递增。

现有 `repair_orders.id` 是全局主键，因此序号采用全系统月度序列，不在企业内分别计数；企业隔离继续由 `company_id` 保证。允许事务失败造成序号空洞，不允许重复或回退。历史网页工单号继续有效，新旧编号可同时存在。

迁移新增 `order_number_sequences(month_key PRIMARY KEY, next_value, updated_at)`。编号通过单条 `INSERT ... ON CONFLICT DO UPDATE ... RETURNING` 原子分配。

### 2.3 幂等和未知结果恢复

请求必须携带客户端生成的 UUID `operationId`。服务端对规范化后的创建命令生成稳定请求哈希，并以现有键：

```text
company_id + actor + CREATE_ORDER + operation_id
```

隔离幂等操作。

迁移扩展 `order_operations` 的短租约信息，用于安全认领和超时恢复：

- 第一次请求认领操作并写入请求哈希；
- 同一 `operationId`、同一哈希已完成时原样返回历史成功响应；
- 同一 `operationId`、不同哈希返回 `409 OPERATION_ID_REUSED`；
- 仍在有效租约内的重复请求返回 `409 OPERATION_IN_PROGRESS`；
- 租约超时后允许比较并交换式接管；
- 正式编号写入操作的 `target_id` 后再创建工单；恢复时优先复用该编号；
- 工单插入与幂等完成回执使用同一 D1 batch，避免工单成功但回执永久未知；
- 若编号分配后、保存 `target_id` 前中断，重试可分配新编号，旧编号形成允许的空洞。

`GET /api/order-operations/create-order/:operationId` 只能查询当前会话、当前企业和当前操作者自己的创建操作。它返回 `pending|completed|failed`；完成时返回同一工单结果。客户端在超时或连接中断后必须先查询该接口，不能生成新的 `operationId` 盲目重建。

## 3. 统一数据契约

### 3.1 创建元数据

`GET /api/order-creation-metadata` 返回：

- 当前角色与公司能力交集；
- 保险公司选项；
- 车辆类型选项；
- 事故类型选项；
- 当前企业可选负责人；
- 预计交车建议值（字段仍允许与现有网页一致的自定义时间文本）；
- 服务端统一默认值和字段长度约束；
- `serverTime` 和契约版本。

固定枚举也由该接口返回，客户端不再分别维护业务文案。进入新增流程时必须在线获取元数据；离线时只能恢复已有草稿，不能创建依赖未知字典的新草稿。

### 3.2 创建命令

新接口接收：

```json
{
  "operationId": "UUID",
  "order": {
    "customer": "客户姓名",
    "phone": "手机号",
    "plate": "车牌号",
    "car": "车型",
    "vin": "可选 VIN",
    "staff": "负责人",
    "insuranceExpiry": "YYYY-MM-DD",
    "insurer": "保险公司",
    "type": "标的车",
    "accidentType": "事故类型",
    "claimNo": "可选报案号",
    "record": "维修记录",
    "laborCents": 0,
    "materialCents": 0,
    "delivery": "待确认",
    "remark": "可选备注"
  }
}
```

金额在线路层统一使用整数分，服务端拒绝负数、非整数和超出安全范围的值。兼容路由把旧网页 decimal 金额严格转换为分，再调用共享服务；服务端响应同时保留当前读取 API 所需的 `laborCents/materialCents/amountCents` 与兼容 decimal 字段，直到所有旧客户端迁移完成。

客户端不发送正式 `id`、日期、时间、企业、角色、状态或版本。服务端按北京时间生成日期和时间，强制：

- `company_id` 为当前会话企业；
- `status = 在修中`；
- `version = 1`；
- 结算、回款、作废和回执字段为空；
- `amount = labor + material`。

### 3.3 校验规则

网页端、Android 和服务端使用同一规则与错误 key：

- 必填：客户、手机号、车牌、车型、保险到期日、维修记录；
- 手机号和车牌本阶段只做去空白后的非空校验，与现有网页规则保持兼容；
- 日期必须是有效 `YYYY-MM-DD`；
- 金额必须非负且来源输入最多两位小数；
- 保险公司、负责人、车辆类型和事故类型必须属于本次元数据返回的允许集合；预计交车允许使用建议值或自定义时间文本；
- 可选文本统一 trim，服务端执行长度上限；
- 未知字段忽略，系统字段即使出现也不采纳。

校验失败返回 `400 VALIDATION_FAILED` 和稳定的字段错误映射。两端把错误显示在对应控件，不依赖解析中文错误句子。

## 4. 四步创建体验

网页端和 Android 使用相同步骤、字段顺序、标题、必填标识和确认摘要：

1. **客户与车辆**：客户、手机号、车牌、车型、VIN、负责人；
2. **保险与事故**：保险到期日、保险公司、车辆类型、事故类型、报案号；
3. **维修与费用**：维修记录、工时费、材料费、交付状态、备注；
4. **确认提交**：完整摘要、总金额、校验结果和提交动作。

正式编号在提交成功前不展示或编辑，只显示“提交后由系统生成”。日期和时间可按服务端时钟预览，但最终以响应为准。

Android 的工作台“新增工单”和五栏中间“新增”进入同一状态持有者。创建成功后：

1. 删除当前本地草稿；
2. 把服务端返回的摘要和详情写入 Room；
3. 刷新 current 列表和工作台指标；
4. 切换到“工单”栏并打开新工单详情；
5. 重置“新增”栏为初始状态。

网页端创建成功后执行同样的服务端详情替换、列表刷新和详情跳转。任何一端都不能把提交前的本地对象当作正式工单插入业务列表。

### 4.1 组件状态

两端均覆盖默认、聚焦、按下、错误、加载和禁用状态。网页端额外覆盖鼠标悬停；Android 按原生语义覆盖触摸按下、焦点和无障碍状态，不伪造移动端 hover。

触控目标不小于 48dp；底部操作区固定提供上一步、下一步/提交和保存草稿。内容区域可滚动并正确避让软键盘。提交中锁定字段与按钮，防止重复操作。

## 5. 草稿与离线边界

每个 `actor + company` 在每个平台只保留一个新增工单草稿。草稿包含本地 UUID、元数据契约版本、表单字段、当前步骤和更新时间，不含正式工单号。

Android 复用 `EncryptedOrderStore` 和 Room `order_drafts`；网页端使用 IndexedDB 保存 AES-GCM 加密载荷，密钥使用 Web Crypto 生成的不可导出浏览器本地密钥，不把手机号或 VIN 明文写入 `localStorage`。

草稿在有意义字段变化后防抖自动保存，并在上一步、下一步、应用进入后台或网页隐藏时立即保存。重新进入新增流程时提供继续或放弃；离开已修改表单时提供保存草稿、放弃、继续编辑。

草稿是明确标注的“未同步本地内容”，不属于正式业务数据。离线时允许继续编辑已有草稿，但禁用正式提交。登出、会话失效或企业切换按现有安全生命周期删除该会话可访问的草稿；提交成功也立即删除草稿。

## 6. 权限模型

本设计不减少用户业务权限，只取消客户端自行指定系统字段的能力。

- `CREATE_ORDER` 决定用户是否能进入并提交新增流程；
- 公司能力开关和角色权限必须同时允许；
- 员工继续要求 `repair` 权限，管理员不能绕过关闭的公司能力；
- 状态推进、编辑、结算、回款、作废和回执维护仍由后续专用命令及各自能力控制；
- 两端均以服务端响应的能力列表显示、禁用或隐藏动作；
- 能力在填写期间被关闭时，提交返回 `403 CAPABILITY_DISABLED`，保留本地草稿并刷新界面权限。

Stage 2 部署完成并通过未认证、授权和隔离门禁后，为目标企业启用 `CREATE_ORDER`。紧急回退只需关闭该能力，不影响读取和本地草稿。

## 7. 错误处理与安全

稳定错误语义：

- `400 VALIDATION_FAILED`：显示字段错误；
- `401 UNAUTHORIZED`：使会话失效并执行现有敏感缓存清理；
- `403 FORBIDDEN/CAPABILITY_DISABLED`：保留草稿、刷新能力；
- `409 OPERATION_ID_REUSED`：阻止提交并生成诊断事件；
- `409 OPERATION_IN_PROGRESS`：进入结果确认流程；
- `5xx` 或网络中断：保留草稿并查询原操作；
- malformed response：不得将本地对象视为成功，进入结果确认流程。

服务端日志和异常不记录手机号、VIN、报案号、备注、完整请求或完整响应。审计记录只保存操作者、企业、动作、目标 ID、安全差异摘要和时间。所有查询与写入都强制使用会话企业，其他企业的工单继续统一表现为不存在。

## 8. 测试策略

### 8.1 服务端与迁移

- migration 合同：月度序列表、幂等租约字段、索引和能力数据；
- 编号：北京时间月份、原子递增、月度切换、空洞允许、历史 ID 共存；
- 创建：服务器字段强制、金额规范化、必填与枚举校验；
- 权限：角色、企业能力和跨企业隔离；
- 幂等：成功重放、哈希冲突、并发处理中、超时接管和未知结果恢复；
- 兼容：旧 `POST /api/orders` 的“新增”分支与新接口进入同一创建服务并产生同一标准结果，既有工单更新分支不被阶段 2 破坏；
- 操作查询：仅本人、本企业、对应动作可见。

### 8.2 网页端

- 四步字段、验证、前后导航和确认摘要；
- 元数据加载、权限禁用和服务端字段错误映射；
- 加密草稿保存、恢复、放弃与登出清理；
- 提交锁、未知结果查询、成功详情与列表刷新；
- 与旧管理、查询和打印功能的回归测试。

### 8.3 Android

- 纯 Kotlin 表单状态、规范化、金额转换和草稿序列化测试；
- 创建 API、操作查询、错误映射和协程取消测试；
- Repository 的 Room 写入、草稿删除、刷新和未知结果恢复测试；
- Compose 四步流程、组件状态、离开确认和成功导航测试源码；
- Android 测试代码编译、Lint 和 APK 构建，不启动模拟器。

### 8.4 双端契约

以共享 JSON fixtures 驱动 Node 与 Android 测试，验证相同业务输入的请求字段、金额分值、默认值、枚举 wire value 和服务端标准响应一致。任何契约字段变更必须同时更新服务端、网页、Android 和 fixtures。

## 9. 发布与回退

发布顺序：

1. 本地 migration 与服务端测试；
2. 远端 D1 备份；
3. 应用新 migration；
4. 部署共享创建服务和兼容路由；
5. 执行未认证、授权、权限和企业隔离冒烟；
6. 部署网页端统一创建流程；
7. 为目标企业启用 `CREATE_ORDER`；
8. 执行网页测试与生产构建；
9. 执行 Android JVM、Android 测试源码编译、Lint 和 APK 构建；
10. 生成可安装 APK 与真机测试清单。

回退优先关闭 `CREATE_ORDER`，使双端回到安全只读状态。数据库迁移只新增表、列和索引，不删除历史字段；旧读取与兼容响应继续工作。远端 D1 备份保留到真机验收完成。

## 10. 完成标准

- 网页端与 Android 可以使用同一账号、同一企业和同一规则创建工单；
- 新工单由服务端生成唯一 `ROYYYYMM#####` 编号；
- 重试和未知网络结果不会产生重复工单；
- 两端正式列表与详情均来自服务端，字段、状态、金额和字典一致；
- 权限关闭、离线、登录失效、校验失败和服务异常均有明确安全行为；
- Node/Vite、Android JVM、Android 测试源码编译、Lint 和 APK 构建全部通过；
- 未启动 Android 模拟器，连接式用例未被错误声明为已执行；
- 设计、计划、交接文档、Git 提交和 GitHub 分支全部同步。
