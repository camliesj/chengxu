# Android 生产客户端基础工程设计

日期：2026-07-16
状态：待用户审阅
适用范围：Android 生产客户端第一阶段

## 1. 目标

在现有网页端、Windows 客户端和移动 UI 图集之外，新建一个独立的 Android 原生工程。第一阶段交付可安装、可启动、可切换五个主导航、可区分管理员与员工、可显示断网只读状态的应用壳层，并实现与已批准图集一致的员工和管理员工作台静态页面。

本阶段不连接真实 API、腾讯云 COS 或 Room 数据库，不实现工单写入。这样可以先验证工程、导航、权限模型和视觉基准，再逐步接入业务数据。

## 2. 技术路线

### 采用：Kotlin + Jetpack Compose

- 只支持 Android，最低 Android 8.0（API 26）。
- 单 Activity、Jetpack Compose、Material 3 基础组件。
- 使用稳定版 Navigation 3 管理类型安全的页面栈和五个主导航。
- 使用 ViewModel、StateFlow 和单向数据流管理 UI 状态。
- 业务数据通过 Repository 接口进入 UI，页面不直接访问网络或本地存储。
- Gradle 版本目录统一固定插件与依赖版本，避免散落版本号。

Android 官方将 Compose、单 Activity、Repository、ViewModel、协程与 Flow、单向数据流列为新应用的推荐架构。本项目采用这些边界，但第一阶段保持单模块，避免过早拆分多模块。

### 未采用：Capacitor + React

虽然能复用网页代码，但移动 UI 已经明确独立设计。继续使用 WebView 会让导航、离线缓存、系统返回手势、图片选择和后续更新行为依赖浏览器层，也会重新引入网页布局耦合。

### 未采用：Flutter

Flutter 也能满足独立 UI，但会引入 Dart 和另一套组件、测试及构建生态。当前项目没有 Flutter 资产，而 Android 是唯一移动平台，因此收益不足以覆盖维护成本。

## 3. 工程边界

Android 工程放在仓库根目录的 `android-client/`，与 `src/` 网页端和 `src-tauri/` Windows 端并列。调试构建暂用 `com.chengxu.autoservice` 作为 `applicationId`，显示名称为“汽修接待”。正式发布前必须结合最终品牌与签名再次确认发布包名；一旦上架，不再修改发布包名。

第一阶段保持一个 `app` 模块，内部按职责分包：

```text
android-client/
  app/src/main/java/com/chengxu/autoservice/
    MainActivity.kt
    AutoserviceApp.kt
    core/designsystem/
    core/model/
    core/network/
    core/session/
    navigation/
    ui/shell/
    ui/workbench/
```

- `core/designsystem`：颜色、字号、间距、圆角、状态组件、按钮和卡片。
- `core/model`：角色、公司、导航项和工作台展示模型。
- `core/network`：只提供网络状态接口及第一阶段的可控假实现。
- `core/session`：当前公司、账号、角色与权限快照。
- `navigation`：类型安全路由、五栏主导航和返回栈规则。
- `ui/shell`：状态栏、离线提示、内容区和固定底部导航。
- `ui/workbench`：员工与管理员共享结构、不同摘要和权限动作。

## 4. 导航与页面状态

主导航固定为：工作台、工单、新增、档案、我的。中间“新增”保持突出圆形图标，但不得改变五栏宽度或导致布局位移。

第一阶段只有工作台是完整静态页面，其余四个导航显示清晰的阶段占位页，不伪装成可用业务功能。每个主导航保留独立返回栈；重复点击当前导航返回该栈根页面。

应用启动先进入本地演示会话，不出现访问码提示。仅调试构建通过测试入口切换演示角色，该入口不得出现在正式发布构建中：

- 员工：显示员工工作台，不显示结算命令。
- 管理员：显示经营摘要和“办理结算”入口。

后续接入登录 API 时，仅替换 SessionRepository，不重写页面权限判断。

## 5. 设计系统

以 `design/mobile-ui/output/` 为唯一视觉基准：

- 背景 `#F5F7FA`
- 表面 `#FFFFFF`
- 主色 `#1677FF`
- 主文字 `#172033`
- 次文字 `#667085`
- 弱文字 `#98A2B3`
- 边框 `#E4EAF2`
- 圆角 `8dp`
- 手机设计基准 `390 × 844`

状态必须同时使用文字和图标，不只依赖颜色。字体使用 Android 中文系统字体，不引入远程字体。页面不使用渐变、装饰性光斑、嵌套卡片或横向滚动。

## 6. 网络与离线规则

第一阶段通过 `NetworkMonitor` 暴露 `StateFlow<ConnectionState>`：

- `Online`：正常显示工作台。
- `Offline`：壳层顶部显示“网络不可用，当前为只读模式”。

离线门禁由统一的 `MutationGate` 判断，不由各页面自行拼接规则。第一阶段没有真实写操作，但“新增”入口在离线时必须禁用，并显示只读原因。后续接入 API 时，所有新增、编辑、状态变更、结算和回执写入都复用该门禁。

## 7. 权限规则

- 员工可查看工单，并在后续阶段把工单推进到在修、完工、待结算。
- 员工不可结算、返结算、作废或维护到账回执。
- 管理员拥有员工权限，并可结算、返结算、作废和维护回执。
- 导出、账号管理、字典管理和操作日志继续仅保留在网页端与 Windows 端。

页面只消费 `PermissionSnapshot`，不根据岗位文字判断权限。权限判断集中在 Session 层，后续云端返回权限勾选时可直接映射。

## 8. 测试与验收

### JVM 单元测试

- 员工权限快照不包含结算、返结算、作废和回执权限。
- 管理员权限快照包含所有高权限操作。
- 离线时 MutationGate 拒绝写操作，在线时允许。
- 五个导航项顺序固定，中间新增项位置固定。

### Compose UI 测试

- 员工工作台显示“今日工作”，且不存在“办理结算”。
- 管理员工作台显示“管理员工作台”“经营摘要”和“办理结算”。
- 点击五个底部导航后，选中状态与页面标题一致。
- 断网状态显示只读提示并禁用新增入口。
- 360dp、390dp 和 412dp 宽度下无横向裁切，长中文文案不覆盖图标或按钮。

### 构建与模拟器

- `assembleDebug` 成功。
- APK 可安装到 Android 模拟器。
- 通过 UI 树定位完成五栏切换、角色切换和离线状态检查。
- 截图与 `workbench-employee.png`、`workbench-admin.png` 进行人工视觉对照。

## 9. 第一阶段完成标准

- `android-client/` 可独立构建，不影响网页端和 Windows 端。
- 最低系统版本为 API 26。
- 五栏导航、主题、员工/管理员工作台和离线只读状态可运行。
- 正式发布构建不包含演示角色切换入口。
- 角色和网络状态由可替换 Repository 驱动，不写死在 Composable 内。
- 单元测试、Compose UI 测试、Debug APK 构建和模拟器验证全部通过。
- 不包含 API、COS、Room、登录、真实工单编辑或发布签名配置。
