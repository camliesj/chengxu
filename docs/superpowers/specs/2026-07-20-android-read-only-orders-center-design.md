# Android 只读工单中心设计

## 背景

Android 客户端已经完成生产认证、公司隔离 Room 缓存、在线刷新、离线只读和真实工作台指标。五栏导航中的“工单”目前仍是阶段占位页，工作台最近工单点击后也只显示“详情将在后续阶段接入”的提示；`AppRoute.OrderDetail(orderId)` 和每个标签独立返回栈已经预留。

本里程碑将现有真实工单快照接入完整的只读列表与详情闭环。用户已确认采用“列表与详情一次完成”的方案，不扩展服务端接口、不修改 Room Schema、不增加敏感字段缓存，也不开放任何业务写入。

## 目标

- 将五栏“工单”占位页替换为真实工单列表。
- 支持本地即时搜索和状态筛选，不为筛选额外请求服务端。
- 统一展示加载、在线刷新、陈旧缓存、离线、空列表和无匹配结果。
- 从“工单”列表和工作台最近工单进入同一个真实详情页。
- 保持每个根标签独立返回栈：从哪个标签进入详情，就返回哪个标签原页面。
- 详情始终从当前 `OrdersRepository.snapshot` 按 `orderId` 查找，缓存更新时同步更新，工单消失时不保留旧对象。
- 继续遵守公司隔离、退出清缓存、401 会话失效、离线只读和无敏感日志约束。

## 非目标

- 不新增、编辑、作废或删除工单。
- 不推进工单状态，不结算或返结算，不上传或查看结算凭证。
- 不新增手机号、VIN、保险公司、理赔号、工时费、材料费、备注等当前 Android 缓存未保存的字段。
- 不修改 `/api/orders`、Cloudflare Functions、D1 表结构、Room Schema 或数据库版本。
- 不实现分页、服务端搜索、后台定时同步或离线写入队列。
- 不修改网页端工单功能。

## 架构与边界

### 会话级工单 ViewModel

新增 `OrdersViewModel`，与现有 `WorkbenchViewModel` 使用同一个认证会话级 `ViewModelStoreOwner`。它直接消费 `OrdersRepository.snapshot`，并持有仅属于当前会话的搜索词和状态筛选。

`OrdersViewModel` 输出单一 `OrdersUiState`，至少包含：

- 全部工单映射后的只读行；
- 当前可见行；
- 搜索词；
- 当前状态筛选；
- 总数与当前结果数；
- 加载、刷新、同步提示和是否可重试；
- 按 ID 查找详情所需的当前工单集合。

退出、会话失效、换账号或换公司时，现有会话级 `ViewModelStoreOwner` 被销毁，搜索、筛选和工单 UI 状态不会跨身份保留。

### 纯映射与过滤

列表和详情显示模型由纯函数从 `RepairOrder` 映射，避免在 Compose 中重复业务判断。搜索先对输入做 `trim` 和大小写归一化，再匹配：

- 工单号 `id`；
- 车牌 `plate`；
- 客户 `customer`；
- 车型 `car`；
- 维修记录 `record`。

搜索与状态筛选取交集。状态固定为“全部 / 在修中 / 已完工 / 待结算 / 已结算”；未知服务端状态不创建动态筛选项，但必须保留原文，并在“全部”结果中正常显示。排序继续使用仓库已提供的日期、时间和稳定 ID 倒序，不在 UI 层重新改变业务顺序。

金额继续使用现有从分到人民币的安全格式化规则；维修摘要优先使用维修记录，其次回退“车型 · 类型”，最后为“暂无维修说明”。

### 导航

`AppNavDisplay` 在 `AppRoute.Orders` 显示真实 `OrdersScreen`，在 `AppRoute.OrderDetail(orderId)` 显示 `OrderDetailScreen`。

- 在“工单”标签点击卡片：对当前“工单”栈执行 `push(OrderDetail(orderId))`。
- 在“工作台”点击最近工单：对当前“工作台”栈执行同一路由压栈。
- 系统返回或详情页返回按钮调用现有 `AppNavigationState.pop()`。
- 再次点击已选中的根标签继续遵循现有规则，清空该标签子栈并返回根页面。

工作台卡片需要把真实 `orderId` 通过回调交给壳层，移除当前 Snackbar 占位提示；不改变快捷操作的权限和离线门禁路由。

## 工单列表界面

### 页面结构

沿用当前品牌设计系统、间距、圆角、颜色和本地 Hugeicons：

1. 页面标题“工单”，辅助信息显示当前结果数量；
2. 全宽搜索框，提示“搜索工单号、车牌、客户”；
3. 可横向滚动的状态筛选条；
4. 轻量同步/陈旧提示和必要的“重新同步”按钮；
5. 纵向工单卡列表或相应空状态。

每张工单卡至少显示：

- 车牌与客户；
- 原始状态和批准的状态色；
- 维修摘要；
- 工单号；
- 日期与时间；
- 当前总金额；
- 预计交车。

整张卡可点击，具有明确的查看图标/语义，触控高度不小于 48dp。搜索框、筛选项、重试和清除筛选也必须满足不小于 48dp 的有效触控尺寸。页面不出现新增、编辑、状态推进或结算按钮。

### 列表状态

- **首次加载且无缓存：** 显示“正在加载工单”。
- **有缓存并刷新：** 保留卡片，显示“正在同步…”。
- **有缓存且刷新失败：** 保留卡片，显示仓库提供的陈旧原因和“重新同步”。
- **无缓存且失败：** 显示错误说明和“重新同步”。
- **真实空列表：** 显示“暂无工单数据”。
- **筛选无结果：** 显示“未找到匹配工单”，提供“清除筛选”；不得误导为公司没有工单。
- **离线：** 沿用壳层全局“网络不可用，当前为只读模式”横幅，继续浏览缓存；主动刷新入口禁用或不显示。

点击重新同步只调用一次现有 `OrdersRepository.refresh()`；重复刷新继续由仓库互斥约束合并。

## 工单详情界面

详情页顶部提供返回按钮、标题“工单详情”和只读状态说明。主要内容为可滚动品牌卡片分区：

### 概览

- 车牌；
- 客户；
- 工单状态。

### 工单信息

- 工单号；
- 接车日期；
- 接车时间。

### 车辆与服务

- 车型；
- 工单类型；
- 维修记录。

### 交付与保障

- 预计交车；
- 保险到期。

### 费用

- 当前总金额。

空字段统一显示“未填写”或与业务语义相符的现有回退文案，不能显示空白占位。未知状态保留原文并使用中性状态色。详情页不展示任何伪写入按钮；离线时内容完全可读。

若当前快照中找不到路由指定的 `orderId`，显示“工单不存在或已失效”和返回入口，不缓存或继续展示刷新前的旧工单对象。若 401 导致会话失效，则沿用认证根切换直接返回登录页。

## 数据流

1. `OrdersRepository` 继续先发布当前公司的 Room 缓存，再在联网时刷新生产 `/api/orders`。
2. `OrdersViewModel` 收集快照并映射全部列表行。
3. 搜索词或状态改变时，只重算本地可见行，不调用仓库刷新。
4. `OrdersScreen` 只消费 `OrdersUiState` 和窄回调：更新搜索、选择状态、清除筛选、刷新、打开详情。
5. `OrderDetailScreen` 接收从当前状态按 ID 查到的显示模型；快照变化会重新查找并更新详情。
6. 工作台最近工单继续由同一仓库快照派生，但点击动作改为导航。

不引入第二份工单数据源，不把 Compose 状态写回 Room，不把 Token、账号或额外敏感字段加入 UI 模型或日志。

## 错误与生命周期

- 网络、服务器和响应异常继续由 `CachedOrdersRepository` 转换为精确同步提示。
- UI 不吞掉或重写仓库的陈旧原因。
- 搜索和筛选在普通刷新期间保留；认证会话变化时随会话 ViewModel 一起清空。
- 401 继续执行“清空公开列表与缓存，再失效认证会话”的既有顺序。
- 协程取消不转换为业务错误。
- 当前详情工单被刷新删除时立即进入失效状态，不显示跨快照旧数据。

## 测试策略

遵循 RED–GREEN–REFACTOR，每个生产行为先由失败测试锁定：

1. **纯函数 JVM 测试：** 工单映射、金额、摘要回退、未知状态、空字段、搜索字段、大小写/空格归一化、状态交集和顺序保持。
2. **ViewModel JVM 测试：** 初始缓存、加载、刷新保留、陈旧提示、空列表、无匹配、搜索/筛选保持、清除筛选、刷新一次和详情工单消失。
3. **导航 JVM 测试：** 从工作台与工单标签分别压入详情、返回各自根页面、根标签重选清栈。
4. **Compose Android 测试源码：** 列表卡片、搜索、筛选、空状态、重试、详情分区、失效状态、返回操作和 48dp 触控尺寸。
5. **生产装配回归：** 同一会话级 owner 创建工作台与工单 ViewModel，回调从屏幕贯通到 `AppNavigationState`，退出时一起清理。
6. **完整验证：** JVM 单元测试、Android 测试代码编译、`lintDebug` 和 Debug APK 构建。

按用户既定要求不启动 Android 模拟器、不执行连接式测试，也不声称 Compose 场景已在设备运行。完成后复制、哈希并验签可安装 APK，由用户在真实手机验证列表密度、搜索输入法、筛选横向滚动、详情返回、离线缓存和生产数据。

## 交付标准

- “工单”根页面不再出现阶段占位文案。
- 工作台最近工单不再显示“详情将在后续阶段接入”。
- 真实列表、搜索、筛选、同步/错误状态和详情闭环均有代码与测试契约。
- 不存在新增、编辑、状态推进、结算或其他写入入口。
- 重要改动后更新 `docs/latest-handoff-prompt.md`、提交并推送当前分支。
